# Internationalization (sq / en)

## Source of truth

- [`messages/sq.json`](../messages/sq.json) — Albanian (default locale, unprefixed URLs)
- [`messages/en.json`](../messages/en.json) — English (`/en/...` prefix)

Client: `useTranslations` / `getTranslations` (next-intl).  
Server: [`src/lib/i18n/server-messages.ts`](../src/lib/i18n/server-messages.ts).

Shop: `/shop` (sq), `/en/shop` (en) via `x-shop-locale` and `NextIntlClientProvider` on the shop layout.

## Conventions

- Do **not** add `locale === "sq" ? ... : ...` for product UI copy.
- Do **not** add `const t = (sq, en) => (en ? e : sq)` helpers.
- Add keys to both `sq.json` and `en.json` in the same namespace.
- Run `npm run i18n:audit` before merging i18n changes.

## Documented exceptions

| String | Where | Rule |
|--------|--------|------|
| **Support ticket** | `nav.createSupportTicket` | Must stay **"Support ticket"** in **both** sq and en (main site header CTA only). |
| Brand terms | SLA, B2B, SKU, COD, IT Arena | May stay identical in both languages. |
| CMS fields | `nameSq` / `nameEn` in site content | Not in message files. |
| User-generated content | Tickets, comments, reports | Not translated by the app. |

Portal/admin ticket labels use normal translations (e.g. sq `Biletat e Mbështetjes`).

## API errors

Routes should return localized `{ error: string }` using `apiErrorResponse(request, "apiErrors.someKey")`.  
Clients pass locale via `Accept-Language`, `?locale=en`, or session context.

## Audit

```bash
npm run i18n:audit
```

Checks sq/en key parity and lists files still using inline bilingual patterns.
